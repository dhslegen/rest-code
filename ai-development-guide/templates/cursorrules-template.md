# .cursorrules 模板示例

> 这是一个高质量的 .cursorrules 配置模板，用于指导Cursor更好地理解项目和生成代码

## 基础模板

```
# 项目上下文
这是一个 [项目类型] 项目，主要功能是 [核心功能描述]。

## 技术栈
- 前端：[技术栈]
- 后端：[技术栈]
- 数据库：[技术栈]
- 其他：[其他重要技术]

## 代码风格
- 使用 [语言版本]
- 遵循 [代码规范，如ESLint配置]
- 缩进：[空格数或Tab]
- 命名规范：
  - 文件名：[如kebab-case]
  - 变量名：[如camelCase]
  - 类名：[如PascalCase]
  - 常量：[如UPPER_CASE]

## 开发规范
1. 所有函数必须有类型注解
2. 复杂逻辑必须添加注释
3. 使用 [测试框架] 编写单元测试
4. 遵循 [设计模式或架构原则]

## 禁止事项
- 不要使用 any 类型
- 不要直接修改原始数据
- 不要忽略错误处理
- 不要使用已弃用的API

## 优先级
- 代码可读性 > 性能优化（除非性能关键）
- 类型安全 > 开发速度
- 可维护性 > 代码简洁性

## 特殊要求
[根据项目具体情况添加]
```

## 实战示例：TypeScript + React项目

```
# AI辅助开发指南项目

## 项目概述
这是一个知识分享文档项目，使用Markdown格式组织内容，涵盖AI辅助开发的完整指南。

## 文档规范
- 使用Markdown格式
- 遵循GitHub Flavored Markdown规范
- 中英文之间添加空格
- 使用相对路径引用资源

## 内容要求
1. 每个章节独立成文件
2. 保持内容的专业性和准确性
3. 提供实际可运行的代码示例
4. 添加必要的图表和截图

## 写作风格
- 简洁明了，避免冗长
- 使用列表和表格组织信息
- 代码块必须指定语言
- 重要概念使用加粗标记

## AI辅助要求
- 生成内容时保持一致的语气
- 技术术语使用准确
- 提供可验证的参考链接
- 示例代码必须可运行

## 文件组织
- `chapters/` 存放各章节详细内容
- `assets/` 存放图片等资源
- `references/` 存放参考资料
- `templates/` 存放可复用模板
```

## 实战示例：Python数据分析项目

```
# Python数据分析项目配置

## 技术栈
- Python 3.10+
- pandas, numpy, matplotlib
- Jupyter Notebook
- pytest

## 代码规范
- 遵循 PEP 8
- 使用 type hints
- 函数和类必须有 docstring
- 最大行长度：88（Black格式化）

## 命名规范
- 文件名：snake_case
- 函数名：snake_case
- 类名：PascalCase
- 常量：UPPER_CASE
- 私有成员：_leading_underscore

## 数据处理规范
1. 使用 pandas 处理结构化数据
2. 数据清洗步骤必须有注释
3. 可视化使用统一配色方案
4. 所有绘图必须有标题和标签

## 错误处理
- 使用具体的异常类型
- 提供有意义的错误信息
- 记录错误日志
- 数据验证使用assert

## 测试要求
- 核心函数必须有单元测试
- 测试覆盖率 > 80%
- 使用 fixtures 管理测试数据
- 测试文件命名：test_*.py

## 性能考虑
- 大数据集使用 chunking
- 避免不必要的数据复制
- 使用向量化操作代替循环
- 必要时使用 numba 加速

## 文档要求
- README 包含快速开始指南
- 复杂算法提供数学公式
- 提供示例 Jupyter Notebook
- API 文档使用 Sphinx 生成
```

## 高级技巧

### 1. 上下文管理
在 .cursorrules 开头提供清晰的项目背景，帮助AI快速理解项目性质。

### 2. 渐进式细化
从整体规范到具体细节，逐层定义约束。

### 3. 示例驱动
提供好的代码示例，AI会模仿这种风格。

### 4. 禁止列表
明确列出不希望AI做的事情。

### 5. 优先级说明
当规则冲突时，说明如何取舍。

---

## 注意事项

1. **不要过度复杂**：规则太多会混淆AI
2. **保持更新**：随项目演进及时更新规则
3. **团队共识**：确保团队成员认同这些规则
4. **测试验证**：新规则加入后观察AI的表现
